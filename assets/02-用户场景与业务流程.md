# 智能客服用户场景与业务流程

> 说明：本文件重点把“用户是谁、为什么来、怎么走到结果”讲清楚，便于后续用于 PRD、面试和作品集。

## 一、用户角色

### 1.1 外部用户

根据现有材料，外部用户主要分两类：

- 船长、船东、货代等航运业务用户。
- 技术人员或企业业务人员，关注平台接入、数据和商务对接。

这类用户的典型特征是：

- 时间紧。
- 目标明确。
- 专业术语多。
- 希望快速得到答案，必要时直接完成操作。

### 1.2 内部用户

内部用户主要是 Hifleet 的客服、运营或销售人员。

他们关注的重点不是“回答得多花哨”，而是：

- 是否能减少重复咨询。
- 是否能准确识别有效线索。
- 是否能在非工作时间承接客户意向。
- 是否能把复杂需求分流给正确的人或系统。

## 二、典型场景

### 2.1 FAQ 场景

用户会问：

- 怎么注册账号？
- 付费版有什么功能？
- 如何管理船队？
- 某个功能在哪里？

这类问题适合知识库问答。

### 2.2 业务操作场景

用户会问：

- 帮我查询某船当前船位。
- 帮我更新船位信息。
- 查询某类档案或业务数据。

这类问题不只是“答复”，还需要触发具体工具或接口。

### 2.3 商务承接场景

用户会表达：

- 试用咨询。
- 购买意向。
- 合作咨询。

这类问题的重点是线索识别、收集和转交，而不是直接生成标准答案。

### 2.4 多渠道接入场景

从现有材料看，项目希望支持多个入口，例如网站、公众号等。其价值不是“多做几个入口”，而是：

- 统一服务能力。
- 统一知识和工作流。
- 不同渠道共用同一套处理逻辑。

## 三、用户链路

### 3.1 标准链路

可以整理成以下流程：

1. 用户从网页、公众号或其他入口发起提问。
2. 系统进行意图识别。
3. 根据意图进入不同分支：
   - FAQ / 知识问答
   - 操作型任务
   - 商务线索承接
   - 无法识别则转人工或兜底
4. 系统返回结果。
5. 如涉及线索或操作，记录到后台或任务流中。

### 3.2 问答链路

适合处理标准化问题：

- 用户提问。
- Prompt 规范问题表达。
- RAG 从知识库检索相关内容。
- 大模型生成回答。
- 低置信度时提示转人工或补充信息。

### 3.3 操作链路

适合处理确定性任务：

- 用户提出动作型需求。
- 系统识别为工具调用。
- 工作流或 Agent 选择对应 API。
- 调用成功后返回执行结果。

### 3.4 线索链路

适合处理商务咨询：

- 系统识别购买或合作意图。
- 收集联系方式、需求类型、场景信息。
- 记录到后台或交给人工销售。

## 四、业务流程图文字版

可以将业务流程抽象为：

```text
用户提问
  -> 意图识别
    -> FAQ / 知识问答
    -> 操作型任务
    -> 商务线索
    -> 兜底转人工
  -> 返回结果 / 记录线索 / 调用接口
```

这个流程的关键不在于“是否能聊天”，而在于：

- 是否能分流。
- 是否能执行。
- 是否能兜底。

## 五、产品判断

### 5.1 为什么多渠道接入重要

多渠道接入的价值不是渠道数量，而是统一用户体验和统一运营逻辑：

- 用户在哪个渠道发起，得到的能力一致。
- 知识库、Prompt、工作流和兜底逻辑共用。
- 便于后续统计各渠道问题类型和转化效果。

### 5.2 为什么不能只做 FAQ

只做 FAQ 会有三个局限：

- 只能回答，不能执行。
- 只能覆盖标准问题，无法承接复杂任务。
- 无法支持商务线索和 7x24 服务目标。

所以该项目必须从“问答”升级到“问答 + 操作 + 承接”的一体化服务。
