# 智能客服技术深挖与产品判断

> 说明：本文件的目标不是堆技术名词，而是把“技术怎么服务产品目标”讲透，便于面试时体现 AI 产品经理需要的技术理解力。

## 一、Prompt 在这里解决什么问题

Prompt 不是装饰层，而是产品控制层的一部分。

它主要解决四件事：

- 统一回答口径，避免模型随意发挥。
- 根据问题类型选择不同处理策略。
- 明确什么时候回答、什么时候检索、什么时候转人工。
- 控制多轮对话中的上下文使用方式。

从产品角度看，Prompt 相当于把“客服规则”显式写进系统。

## 二、RAG 为什么需要

如果没有 RAG，大模型只能依赖自身通用知识，面对企业产品问题时容易出现：

- 回答不准确。
- 回答过于泛化。
- 回答与企业规则不一致。

RAG 的价值是把企业知识接入回答链路，让模型“先查再答”。

### 2.1 知识库如何进入问答链路

一个合理的链路可以这样理解：

1. 用户提出问题。
2. 系统对问题进行清洗和意图判断。
3. 在知识库中检索相关文档片段。
4. 将检索结果与用户问题一起输入模型。
5. 模型基于检索内容生成回答。
6. 如果检索置信度不够，提示转人工或继续补充信息。

### 2.2 产品判断

对于智能客服来说，RAG 不是可选项，而是提高可控性和准确性的关键手段。尤其在企业知识问答中，RAG 比“直接让模型回答”更符合产品要求。

## 三、工作流 / Agent 和普通问答的区别

### 3.1 普通问答

普通问答解决的是：

- 用户问了什么。
- 系统给出什么答案。

它适合 FAQ、规则说明、知识解释。

### 3.2 工作流

工作流解决的是：

- 这个任务需要经过哪些固定步骤。
- 每一步调用什么工具。
- 每一步失败后怎么兜底。

它适合确定性高的任务，例如查询船位、更新信息、收集线索。

### 3.3 Agent

Agent 更强调动态规划和工具选择：

- 先理解目标。
- 再决定调用哪些工具。
- 根据中间结果继续调整。

它适合更开放、步骤不完全固定的任务。

### 3.4 产品判断

在这个项目中，更建议采用“工作流优先，Agent 兜底”的思路：

- 确定性任务尽量工作流化。
- 不确定任务再交给 Agent。

这样更稳定，也更容易验收。

## 四、COZE、Ollama、Milvus、LangChain 的角色

### 4.1 COZE

COZE 更像产品验证和快速交付平台：

- 适合快速搭建交互。
- 适合验证 Prompt、插件和工作流。
- 适合展示早期产品形态。

### 4.2 Ollama

Ollama 更像本地模型容器：

- 适合私有化和本地验证。
- 适合探索企业场景中的部署可行性。

### 4.3 Milvus

Milvus 是知识检索层：

- 负责向量存储。
- 负责召回相关知识片段。
- 直接影响 RAG 的效果。

### 4.4 LangChain

LangChain 更像链路编排层：

- 负责把模型、检索、工具和记忆串起来。
- 适合做复杂链路验证。

## 五、公有云与私有化的产品判断

### 5.1 公有云适合什么

- 早期验证。
- 快速上线。
- 低成本试错。

### 5.2 私有化适合什么

- 企业知识敏感。
- 部署环境受限。
- 对数据控制要求高。

### 5.3 最终判断

对于这个项目，不应一开始就强行做全私有化，而是分阶段推进：

- 先验证产品闭环。
- 再验证知识效果。
- 再验证私有化可行性。

这比一上来就追求“最完整架构”更符合产品化节奏。

## 六、产品边界与技术边界

### 6.1 产品边界

智能客服产品要解决的是：

- 常见问题自动答复。
- 标准任务自动执行。
- 线索自动承接。

### 6.2 技术边界

技术系统不能替代的部分包括：

- 复杂业务决策。
- 高风险操作的最终确认。
- 知识不充分时的强行回答。

所以必须保留：

- 置信度阈值。
- 人工兜底。
- 操作确认。

## 七、可讲的产品判断点

面试时最值得强调的判断是：

- 为什么不直接做纯聊天，而是做“问答 + 工作流 + 线索承接”。
- 为什么先用 COZE 验证，再考虑私有化。
- 为什么知识库要接入 RAG，而不是只靠 prompt。
- 为什么确定性任务更适合工作流，而不是完全交给 Agent。
