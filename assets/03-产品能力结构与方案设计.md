# 智能客服产品能力结构与方案设计

> 说明：本文件重点从产品结构角度拆解“这个智能客服应该长什么样”，同时保留必要的技术映射，方便后续写简历、讲面试和做 GitHub 展示。

## 一、产品目标

这个项目的产品目标可以概括为三句话：

- 把高频知识咨询自动化。
- 把可标准化操作工作流化。
- 把客户意向承接成可跟进线索。

## 二、能力结构

### 2.1 核心能力层

可以把产品能力拆成五层：

1. 入口层
   - 网站
   - 公众号
   - 其他业务入口

2. 理解层
   - 意图识别
   - 问题分类
   - 关键实体抽取
   - 多轮上下文理解

3. 知识层
   - FAQ 知识库
   - 产品手册
   - 业务文档
   - 兜底答案

4. 执行层
   - 工作流
   - Agent
   - API 调用
   - 线索记录

5. 运营层
   - 统计分析
   - 告警和失败兜底
   - 人工接管
   - 知识更新

### 2.2 能力模块拆解

#### 模块 1：私有知识库问答

功能目标：

- 回答注册、功能使用、产品规则等标准问题。

产品价值：

- 降低客服重复劳动。
- 提升用户自助解决率。

#### 模块 2：多轮上下文对话

功能目标：

- 支持追问和澄清。
- 让用户不用重复输入背景信息。

产品价值：

- 降低交互摩擦。
- 提高复杂问题的可完成率。

#### 模块 3：任务型工作流

功能目标：

- 将“查询船位、更新船位”等动作包装成可配置任务。

产品价值：

- 从“回答问题”升级到“完成任务”。

#### 模块 4：商务线索承接

功能目标：

- 识别购买意向。
- 收集线索信息。
- 转交后台或人工。

产品价值：

- 降低机会流失。
- 将客服入口变成销售入口。

#### 模块 5：多渠道统一接入

功能目标：

- 支持网页、公众号等多入口接入。

产品价值：

- 降低维护成本。
- 统一服务口径。

## 三、方案设计

### 3.1 MVP 范围

MVP 不应一开始就追求“全自动客服”，而应优先覆盖三类高价值能力：

- FAQ 自动问答。
- 关键操作型任务。
- 商务线索收集和转交。

MVP 的判断标准是：

- 是否能覆盖高频问题。
- 是否能降低人工重复工作。
- 是否能验证 Agent / 工作流价值。

### 3.2 第二阶段能力

在 MVP 之后，再考虑：

- 更细的意图识别。
- 更复杂的会话记忆。
- 更多接口型任务。
- 更完整的后台运营看板。
- 自动知识更新和评估体系。

### 3.3 为什么这样设计

这种设计逻辑是从“低风险高频”开始：

- 先做标准问答，风险最低。
- 再做确定性操作，价值更高。
- 最后做线索承接，贴近业务结果。

这样可以避免一开始就把系统做成复杂但难验证的“大而全 Agent 平台”。

## 四、技术角色映射

### 4.1 Prompt

Prompt 在这里的作用不是“让模型说得更像人”，而是：

- 约束回答风格。
- 明确回答边界。
- 控制任务分流规则。
- 指导模型在不同意图下采取不同策略。

### 4.2 RAG

RAG 的作用是把平台知识接入大模型：

- 用户提问后先检索相关文档。
- 再基于检索结果生成回答。
- 减少模型胡编。
- 让回答与企业知识对齐。

### 4.3 Agent / 工作流

Agent 和工作流的作用是把“说”变成“做”：

- 普通问答回答的是“是什么”。
- Agent / 工作流执行的是“帮我做什么”。

在该项目里，工作流更适合确定性强的业务动作，Agent 更适合意图不完全固定、需要动态判断的场景。

### 4.4 COZE

COZE 更适合承担快速搭建和验证的角色：

- 低代码搭建智能体。
- 编排 Prompt、插件、知识库和工作流。
- 快速验证产品可行性。

### 4.5 Ollama

Ollama 在方案中承担本地模型运行与私有化验证角色：

- 支持本地部署和脱敏验证。
- 便于探索企业私有化场景。

### 4.6 Milvus

Milvus 主要承担向量知识库能力：

- 存储知识向量。
- 支持相似度检索。
- 服务 RAG 检索链路。

### 4.7 LangChain

LangChain 适合承担链路编排和组件连接：

- 串联模型、检索、工具调用和记忆。
- 用于验证更灵活的私有化实现路径。

## 五、公有云与私有化方案权衡

### 5.1 公有云方案

优点：

- 上手快。
- 验证快。
- 适合 MVP 和快速试错。

缺点：

- 数据和知识的可控性较弱。
- 对企业敏感知识的适配有限。
- 长期成本和依赖外部平台的风险更高。

### 5.2 私有化方案

优点：

- 数据更可控。
- 更适合企业知识和敏感业务。
- 便于后续深度定制。

缺点：

- 部署成本更高。
- 需要更多模型、检索和服务编排能力。
- 交付复杂度更高。

### 5.3 取舍结论

结合现有材料，这个项目更合理的路径是：

- 先用 COZE 做快速验证。
- 再用 Ollama + Milvus + LangChain 探索私有化能力。
- 最终根据数据安全、部署成本和交付要求决定是否切换。
