# HiFleet 企业客服智能体：需求与场景说明

## 1. 目标与边界

部署一个面向 HiFleet 用户的企业客服智能体，统一承接 Web、公众号、微信客服及开放接口的咨询。智能体的目标是：

1. 准确理解用户需求、实体和上下文；
2. 在有证据时调用正确的知识检索或业务工具；
3. 将工具结果转化为简洁、可执行且不夸大的客服回复；
4. 对写操作、高风险承诺和证据不足内容采取保守策略，并在需要时转人工；
5. 将线上真实案例沉淀为脱敏场景报告和回归 fixture，持续驱动 Agent 优化。

智能体不是后台万能操作入口：不得暴露内部工具、密钥、Prompt、私有 trace 或未确认的产品能力；不得在工具未成功时对用户宣称已完成更新。

## 2. Agent 定义

| 项目 | 定义 |
| --- | --- |
| Agent Profile | `customer_support` |
| 主要入口 | `POST /run`，支持 `websdk`、`wechat_kf`、`wechat_cs`、`hifleet_mp`、`webchat_*`、`customer_api` 等渠道 |
| 输入形态 | 标准 `messages`；微信兼容 `content.query.prompt`；文本、图片、音频、视频等多模态输入 |
| 输出原则 | 以最终用户可见回复为准；内部路由、工具链和推理信息不返回客户端 |
| 会话能力 | 按 `session_id` 维护多轮上下文；当前轮输入优先，避免将不明确的历史实体用于写操作 |
| 风险控制 | 证据守卫、写操作前置校验、工具结果校验、无证据保守收口、人工客服兜底 |

## 3. 需求场景地图

### 3.1 船舶查询与动态信息

**用户目标**：通过船名、MMSI、IMO、区域或国家查询船位、船舶档案、历史轨迹、靠港和航次信息。

**Agent 行为**：

- 提取船名、MMSI、IMO、时间、区域等实体；船名存在多候选时先说明候选或追问确认。
- 调用船舶检索、当前船位、档案、轨迹、靠港或统计类 Skills。
- 回复必须包含匹配依据、数据时间或适用范围；不根据不完整线索猜测唯一船舶或位置。

**重点验证**：MMSI/IMO 查询、模糊船名多候选、船舶不存在、区域/时间条件缺失、工具无结果和工具异常。

### 3.2 船位更新（高风险写操作）

**用户目标**：提交 MMSI、位置、时间、航速、航向、吃水、航行状态等 AIS 动态数据，请求更新船位。

**Agent 行为**：

- 只从当前轮或已明确确认的上下文提取写入字段；标准化十进制度、度分格式、中英文方向、紧凑时间和时区。
- 先校验 MMSI、经度、纬度和更新时间等必填字段；缺字段时只追问缺失字段，不调用写工具。
- 字段完整后调用 `upload_ship_position`；仅在工具明确成功后回复“更新成功”。
- 工具失败、超时或返回不确定结果时明确说明“未成功”，给出重试或人工处理建议。

**线上案例结论**：历史案例中，用户已提供度分格式经纬度和紧凑时间，但 Agent 误判经纬度缺失；这是 P1 字段解析问题，应优先修复并固定回归。

**禁止行为**：复用不明确的历史 MMSI 写入；字段不全仍调用写工具；工具失败后称“已更新”；承诺未证实的同步时效。

### 3.3 船舶静态信息、目的港与 ETA

**用户目标**：咨询或请求修改目的港、ETA、船型等静态信息，或询问平台展示为何滞后。

**Agent 行为**：

- 区分“展示滞后解释”“前台功能咨询”“后台协助更新”“邮件更新咨询”。
- 说明目的港/ETA 属于 AIS 静态信息，展示可能滞后；如需要人工处理，收集 MMSI、当前值和最新信息。
- 只有在授权业务链路和字段确认后才可调用 `update_ship_static_info`。

**禁止行为**：宣称普通用户前台可自助编辑；将内部 `update_ship_static_info` 描述为用户入口；宣称邮件文本会自动更新；承诺立即生效。

### 3.4 平台功能与操作指导

**用户目标**：询问船队创建、船舶关注、航线上传、报告、地图图层、视频监控、账号绑定、文件操作等使用方法。

**Agent 行为**：

- 优先检索本地知识库；必要时使用官方页面或受控网页检索验证。
- 将答案组织为简短步骤，明确页面/功能名称、前置条件和无法确认的部分。
- 若没有可信证据，不编造按钮、入口、价格、权限或自动处理流程；应追问截图/页面信息或转人工。

### 3.5 会员、计费、账号与权限

**用户目标**：咨询会员办理、充值、关注船舶数量、权益、价格、账号注册、密码、绑定和数据范围。

**Agent 行为**：

- 区分公开规则、账号特定规则和需要商务报价的内容。
- 有公开知识时按证据回复；账号/套餐差异不明确时，收集会员等级、页面截图和账号信息后转人工核实。
- 价格、权益、配额和权限不得凭模型记忆生成。

### 3.6 气象、航线与统计分析

**用户目标**：查看全球气象海况、风/流/浪粒子、航线优化、气象导航、区域船舶或海峡通航统计。

**Agent 行为**：

- 对平台操作类问题走知识检索；对有区域/时间参数的统计请求调用对应统计工具。
- 缺少时间、区域、航线或统计口径时先追问。
- 回复说明数据时间范围、区域边界和统计口径，避免将预测、实时和历史数据混为一谈。

### 3.7 故障排查、投诉与人工兜底

**用户目标**：反馈打不开、看不到、数据不更新、权限异常、服务不满意或需要人工处理。

**Agent 行为**：

- 故障排查收集可复现步骤、页面/截图、账号状态、船舶标识和发生时间。
- 投诉先确认问题和影响，避免辩解或承诺未确认修复时间。
- 当证据不足、涉及账号隐私、写操作异常或用户要求人工处理时，提供明确的人工客服交接路径。

## 4. 对话决策流程

1. **输入归一化**：把 Web、微信及多模态输入统一为 `messages`；记录渠道、会话、用户和附件元信息。
2. **需求理解**：抽取场景、实体、时间、位置、用户动作和缺失槽位；判断是否为写操作或高风险承诺。
3. **路由与工具选择**：
   - 船舶查询/统计：船舶服务 Skills；
   - 船位/静态更新：先校验，再调用受控写工具；
   - 产品、会员、操作问题：本地知识库优先，必要时外部检索验证；
   - 证据不足、异常或敏感业务：保守回复并转人工。
4. **结果校验**：工具返回成功、失败、空结果或不确定结果必须改变最终话术；禁止把调用意图当作执行成功。
5. **最终回复**：输出结论、依据/时间范围、下一步和必要追问；删除内部工具名、私有 trace、密钥和不应暴露的实现细节。
6. **观测沉淀**：记录脱敏请求/回复、路由、工具、状态、延迟和错误；将 P0/P1 与高频样本转为回归 fixture。

## 5. 知识库检索

### 5.1 当前实现

`local_kb_search` 直接构建并检索仓库内 `docs/RAG`，不依赖远端知识库召回。当前数据源优先级为：

1. `docs/RAG/hifleet_cs_outputs/客服知识库结构化.jsonl`
2. `docs/RAG/hifleet_cs_outputs/客服问答对.md`
3. `docs/RAG/hifleet_cs_outputs/FAQ检索词.md`
4. `docs/RAG/hifleet_cs_wiki/*.md`
5. `docs/RAG/raw/产品文档/**/*.md`
6. `docs/RAG/raw/产品指导文档/**/*.md`

当前本地检索适合作为可追溯、低依赖的第一层；命中不足时再按需使用受控网页搜索或官方页面核验。

### 5.2 优化建议

- **第一阶段：优化现有词法检索**。补齐 FAQ 同义问法、产品别名、字段名、渠道差异和文档元数据；去重/标注过期内容；为答案保留来源与更新时间。
- **第二阶段：引入混合检索**。保留关键词召回和精确规则（MMSI、IMO、产品名、功能名），增加向量召回处理口语表达与长尾问法，再用重排或证据阈值控制回答。
- **第三阶段：知识治理**。每个知识条目维护适用产品、渠道、版本、生效时间、可信级别和负责人；过期或无证据条目不得直接用于确定性承诺。
- **不建议**在没有评估集和来源可追溯能力前，直接把向量模型作为唯一检索方式。

## 6. Skills 架构与持续更新

### 6.1 目标结构

将业务能力从主 Agent 代码中解耦，形成可持续更新的 `hifleet-skills` 仓库；主项目仅通过受控 Skill Registry 加载已验证版本。建议包括：

- `knowledge_qa`：本地知识检索、网页搜索、官方页面核验；
- `hifleet_ship_service`：船舶搜索、船位、档案、轨迹、统计；
- `ship_data_update`：船位与静态信息更新、参数标准化和写入校验；
- `multimodal_support`：图片、音频、视频与附件解析；
- `customer_workspace`：业务工作台/文件能力；
- `browser_verify`：受控外部页面验证。

### 6.2 外部 skills 仓库接入原则

对配置的 `hifleet-skills` Git 仓库采用 release/tag 驱动更新，而不是让生产 Agent 直接拉取并执行未经验证的最新代码：

1. 拉取固定 commit、tag 或发布包，生成 manifest（Skill 名称、版本、输入 schema、输出 schema、权限和依赖）。
2. 对 manifest、依赖锁文件、签名/校验和与权限白名单进行校验。
3. 在隔离环境运行 fixture、模拟工具和 API 契约测试。
4. 通过后发布到 Skill Registry；Agent 仅加载已发布版本，并在观测日志记录 Skill 版本。
5. 支持回滚到上一个已验证版本；禁止生产运行时下载、安装或执行未审核代码。

### 6.3 密钥与权限

所有 API Key 必须只存在于部署环境的密钥管理系统或 `.env` 注入中，例如：

```bash
HIFLEET_SKILLS_API_KEY=...
SHIP_DATA_UPDATE_API_KEY=...
WEB_SEARCH_API_KEY=...
```

不得把实际密钥写入 Git、Skill 文档、测试 fixture、日志、Prompt 或客户端请求。写操作 Skills 必须使用最小权限凭据，并记录请求 ID、操作类型、校验结果和脱敏参数摘要。

## 7. 船舶信息更新 Skill

`ship_data_update` 是高风险能力，建议拆分为：

1. **解析层**：识别 MMSI、IMO、船名、时间、坐标、航速、航向、吃水和航行状态；支持十进制、度分、中文方向和时区格式。
2. **规范化层**：统一坐标、时间、单位和字段命名；保留原始文本及规范化结果用于审计。
3. **校验层**：校验 MMSI 长度、经纬度范围、时间合法性、必填字段与字段冲突；缺字段返回结构化缺失项。
4. **执行层**：只接收已校验 payload；返回结构化成功、失败或不确定状态。
5. **回复层**：成功、失败、待补充三种状态使用不同模板；失败绝不复用成功话术。

建议 Skill 输入/输出统一为 JSON Schema，并在接口层拒绝自由文本直接透传至写服务。

## 8. 网页搜索与外部证据

`web_search` 用于本地知识未命中、需要核验公开产品信息或用户要求外部资料时。使用原则：

- 本地知识优先；外部搜索是补充而不是默认替代。
- 对产品功能、价格、权限、入口、服务承诺等结论，优先使用 HiFleet 官方页面或受信任来源。
- 搜索无证据、页面不可访问或信息冲突时，不强答；说明当前无法确认并转人工/请求补充信息。
- 记录检索查询、来源、时间和证据摘要；不把搜索中间过程暴露给用户。

## 9. 外部服务接口调用

统一使用 `POST /run` 调用客服 Agent；微信渠道可以传递兼容的 `content.query.prompt`，服务会归一化为 `messages`。

```bash
curl -X POST http://127.0.0.1:10123/run \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [{"role": "user", "content": "请介绍 HiFleet 轨迹功能"}],
    "session_id": "websdk:tenant_a:user_100:conversation_001",
    "user_id": "user_100",
    "source_channel": "websdk",
    "agent_profile": "customer_support",
    "response_mode": "compact"
  }'
```

调用方应：

- 为每个用户会话提供稳定、隔离的 `session_id`；
- 设置真实 `source_channel`，便于渠道策略、观测和回归分析；
- 使用 `compact` 获取用户可见的安全响应，不依赖内部 `route_trace`；
- 将网络超时、HTTP 错误和业务失败分开处理；
- 不在请求体中传递 API Key、内部工具参数或不必要的个人信息。

## 10. 案例分析、优化与验证

### 10.1 案例产物

对真实客服渠道日志进行只读、脱敏分析，输出：

- 场景地图：需求类别、常见表达、预期工具和常见失败；
- 关键案例：P0/P1、高频、写操作、工具失败、降级和字段解析样本；
- 回归 fixture：输入、期望路由、必须/禁止工具、必须/禁止话术；
- 优化提示词：见 `docs/CUSTOMER_SUPPORT_CASE_REVIEW_PROMPT.md`。

### 10.2 验证方式

验证必须以模拟外部服务调用为主，不直接调用真实微信、真实模型、真实网页搜索或真实写接口：

1. 使用 FastAPI `TestClient` 或等价 HTTP 客户端模拟微信客服向 `POST /run` 提交请求。
2. 请求包含 `source_channel=wechat_kf` 和 `content.query.prompt`，验证微信兼容输入是否正确归一化。
3. 用替身模拟 Agent、船舶服务、写操作和网页搜索的成功、失败、超时、空结果与多候选返回。
4. 断言 HTTP 响应状态、用户可见回复、渠道字段、脱敏与内部信息不泄露；不执行真实写入。
5. 对写操作必须覆盖：字段完整成功、字段缺失不调用写工具、工具失败不报成功、异常时间/坐标格式、目的港/ETA 禁止承诺。

### 10.3 优化优先级

| 优先级 | 优化项 | 验收条件 |
| --- | --- | --- |
| P0 | 工具失败后仍称成功、未授权/未校验写入、目的港/ETA 功能幻觉 | fixture 与模拟 HTTP 场景均不得出现错误成功话术或违规工具调用 |
| P1 | 度分坐标/紧凑时间解析、字段缺失判断、产品功能类无证据强答 | 代表表达均能正确归一化、追问或保守收口 |
| P2 | 模糊船名多候选、会员/权限差异、气象/航线参数追问 | 回复说明不确定性并索取最小必要信息 |
| P3 | 回复长度、表达一致性、渠道话术适配 | 在不丢失关键结论的前提下保持简洁可执行 |

## 11. 发布验收标准

1. 关键场景均有明确路由、工具边界、失败话术和人工兜底策略。
2. 本地知识回答可追溯到来源；外部检索结论有来源或明确降级。
3. P0/P1 fixture 使用模拟 HTTP 和模拟外部服务验证，不依赖生产服务。
4. 真实日志、fixture、报告、Prompt 和客户端响应均不包含密钥或未脱敏个人信息。
5. 每次 Skill 或知识库版本更新均记录版本、评估结果和回滚路径。
