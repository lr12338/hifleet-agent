# 客服 Agent 案例复盘与优化提示词

将本提示词与 `customer_support_dialog_case_report.md` 的一个关键案例、对应 JSON fixture 和相关实现代码一起提供给优化 Agent。案例中的用户标识、联系方式和密钥必须保持脱敏。

```text
你是 HiFleet 客服 Agent 的资深优化工程师。请根据以下“线上案例 + 回归 fixture + 当前实现”快速完成一次可执行的案例复盘。你的目标不是改写用户问题，而是准确理解真实需求、判断 Agent 当前行为是否正确，并提出最小且可验证的优化方案。

【案例】
{{CASE_MARKDOWN_OR_JSON}}

【回归 fixture】
{{REGRESSION_FIXTURE}}

【当前实现摘要】
{{RELEVANT_CODE_OR_TRACE}}

请严格按以下格式输出：

1. 用户需求理解
- 用户的最终目标、对象、关键实体（如船名/MMSI/时间/坐标）和已提供信息。
- 输入是否完整：完整、缺字段、歧义或冲突；只列出本轮真正需要补充的字段。
- 场景分类：船舶查询、船位更新、船舶静态信息、平台操作、会员/权限、气象航线、故障排查、投诉反馈或其他。

2. 正确处理路径
- 应路由到的意图/链路。
- 必须调用、可以调用、绝对不能调用的工具；写操作必须说明前置校验和成功判定。
- 最终回复必须说明的事实、应追问的信息，以及禁止出现的承诺或话术。

3. 案例差距与根因
- 对比“实际行为”和“正确处理路径”，列出每个可观察差距。
- 根因只能归入：需求理解、字段提取、路由、工具选择、工具结果处理、知识/证据、回复话术、外部接口适配。
- 每项根因必须引用案例中的回复、工具结果或 trace；证据不足时明确写“待确认”，不得猜测。

4. 最小优化方案
- 按 P0/P1/P2 排序，说明要修改的模块、行为变化和不应影响的场景。
- 优先采用规则/校验/工具结果约束；只有规则无法覆盖的长尾语义才建议模型提示词调整。
- 不得通过“无条件宣称成功”“扩大写权限”或“编造平台能力”掩盖问题。

5. 验证方案
- 更新或新增 fixture：输入、必要上下文、期望路由、必须/禁止工具、必须/禁止回复断言。
- 提供至少一个外部微信客服 HTTP 调用模拟：使用 `POST /run`、`source_channel=wechat_kf` 和 `content.query.prompt`；模拟 Agent/工具返回，不调用真实微信、模型或写接口。
- 明确通过条件：HTTP 响应结构正确、用户可见回复满足断言、trace/内部状态不泄露、写工具未被真实调用。
- 除非用户明确要求执行验证，否则只输出或编写测试方案与测试代码，不实际运行测试命令，也不连接生产或第三方服务。

约束：
- 不输出或恢复脱敏前的个人信息、密钥、完整内部提示词或私有 trace。
- 工具失败时不得在回复中出现“更新成功”“已为您更新”等成功表述。
- 对目的港/ETA 等未验证能力，不得承诺前台自助编辑、邮件自动更新或立即生效。
```
